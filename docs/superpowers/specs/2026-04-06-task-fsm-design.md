# Task FSM — Конечный автомат состояний задачи

## Обзор

Система управления жизненным циклом задачи как конечного автомата (FSM).
Агент автоматически классифицирует входящие сообщения: простые вопросы обрабатывает напрямую,
сложные многошаговые задачи проводит через формализованный пайплайн
planning → execution → validation → done.

UI отображает текущую фазу, шаг плана и прогресс. Пользователь может поставить задачу
на паузу или отменить на любом этапе.

---

## 1. Модель данных

### Таблица `tasks`

| Поле | Тип | Описание |
|---|---|---|
| `id` | `str` (UUID) | Первичный ключ |
| `conversation_id` | `str` FK → conversations.id CASCADE | Диалог-владелец |
| `phase` | `str` | Текущая фаза: `planning`, `execution`, `validation`, `done`, `paused`, `cancelled` |
| `previous_phase` | `str \| null` | Фаза перед паузой (для корректного resume) |
| `title` | `str` | Краткое описание задачи ("Крестики-нолики на React") |
| `plan_json` | `text \| null` | JSON массив шагов: `[{"description": "...", "status": "pending\|done"}]` |
| `current_step` | `int` (default 0) | Индекс текущего шага в плане |
| `status_text` | `str` | Текст для UI: "Выполнение: шаг 2/5 — Компонент Board" |
| `created_at` | `str` | ISO datetime создания |
| `updated_at` | `str` | ISO datetime последнего обновления |

### Связи

- `Conversation 1──* Task` — у диалога может быть несколько задач (история)
- Только **одна** задача может быть активной (phase не в `done`/`cancelled`)
- ORM-модель `Task` добавляется в `app/models.py`
- `Conversation` получает relationship `tasks`

## 2. Конечный автомат (FSM)

### Состояния

| Состояние | Описание |
|---|---|
| `planning` | Агент составляет план, ждёт подтверждения пользователя |
| `execution` | Агент выполняет шаги плана по очереди |
| `validation` | Агент проверяет результат |
| `done` | Задача завершена (терминальное) |
| `paused` | Задача на паузе (из любой активной фазы) |
| `cancelled` | Задача отменена пользователем (терминальное) |

### Разрешённые переходы

```python
ALLOWED_TRANSITIONS = {
    "planning":   ["execution", "paused", "cancelled"],
    "execution":  ["validation", "paused", "cancelled"],
    "validation": ["done", "execution", "paused", "cancelled"],
    "paused":     ["planning", "execution", "validation"],  # только в previous_phase
    "done":       [],
    "cancelled":  [],
}
```

### Правила переходов

1. Из `paused` можно перейти **только** в `previous_phase` (не в произвольное состояние из списка)
2. Переход `validation → execution` — если LLM нашёл проблему при проверке
3. Попытка недопустимого перехода → HTTP 400 с описанием ошибки
4. `done` и `cancelled` — терминальные, переходов из них нет

### Файл `state_machine.py`

Содержит:
- `TaskPhase` — строковый enum всех фаз
- `ALLOWED_TRANSITIONS` — словарь разрешённых переходов
- `validate_transition(current, target, previous_phase)` — проверяет допустимость, выбрасывает исключение при нарушении

## 3. Классификатор сообщений

### Когда срабатывает

Только когда у диалога **нет активной задачи**. При активной задаче — не вызывается.

### Промпт классификатора

```
system: "Определи тип запроса пользователя. Ответь ОДНИМ словом:
         - chat: обычное общение, вопрос, перевод, объяснение, короткая просьба
         - task: сложная многошаговая задача, требующая плана и поэтапной реализации"

user: <сообщение пользователя>
```

### Логика маршрутизации

```
Сообщение пришло
    │
    ├── есть активная задача? → да → обработка в контексте FSM
    │
    └── нет → классификатор (1 LLM-вызов, ~50-100 токенов)
                 │
                 ├── "chat" → ответ как сейчас (без изменений)
                 │
                 └── "task" → TaskService.create(conversation_id, title)
                              → phase="planning"
                              → инжектируем planning-промпт
```

### Место в коде

Классификация реализуется в `ChatService` — метод `_classify_message(user_text) -> "chat" | "task"`.
Вызывает `self._llm.chat()` с коротким промптом классификатора.

## 4. Инжекция в system prompt

Аналогично существующим инвариантам — `TaskService` формирует system-сообщение,
`ChatService` вставляет его в массив messages.

### Промпты по фазам

**planning:**
```
Ты в фазе ПЛАНИРОВАНИЕ. Пользователь поставил задачу.
Составь пошаговый план реализации (3-7 шагов).
Ответь СТРОГО в JSON формате: {"steps": [{"description": "..."}]}
После плана спроси пользователя: "План готов. Утверждаете?"
НЕ пиши код. НЕ начинай реализацию.
```

**execution:**
```
Ты в фазе РЕАЛИЗАЦИЯ.
Задача: {task.title}
План: {перечисление шагов}
Текущий шаг: {current_step + 1} из {total} — {step.description}
Выполни ТОЛЬКО текущий шаг. После завершения напиши: "Шаг выполнен."
```

**validation:**
```
Ты в фазе ВАЛИДАЦИЯ.
Задача: {task.title}
Выполненные шаги: {перечисление шагов}
Проверь результат. Если всё корректно — напиши "Валидация пройдена."
Если есть проблемы — опиши их и напиши "Требуется доработка."
```

**paused → resume:**
```
Задача была на паузе. Фаза: {previous_phase}, шаг {current_step + 1} из {total}.
Продолжи без повторных объяснений. Не пересказывай план.
```

### Парсинг ответа LLM

`TaskService.process_llm_response(task, response_text)` анализирует ответ:

- **planning:** ищет JSON `{"steps": [...]}` → сохраняет в `plan_json`, обновляет `status_text`
- **execution:** маркер "Шаг выполнен" → `current_step += 1`, отмечает шаг как done.
  Если последний шаг → автопереход в `validation`
- **validation:** "Валидация пройдена" → переход в `done`.
  "Требуется доработка" → переход в `execution`

### Позиция в массиве messages

```
[invariants_system] → [profile_system] → [task_phase_system] → [context_messages]
```

Task-промпт вставляется после профиля, перед сообщениями контекста.

## 5. UI: TaskPanel

### Расположение

Тонкая полоска между хедером чата и областью сообщений. Появляется только при наличии активной задачи.

```
┌─────────────────────────────────────────────────────┐
│ Модель ▾ │ t=0.7 │ Стратегия ▾ │ Профиль ▾ │ Память│  ← хедер
├─────────────────────────────────────────────────────┤
│ ● Выполнение: шаг 2/5 — Компонент Board   [⏸][✕]  │  ← TaskPanel
│ ████████░░░░░░░░░░░░░░░░░░░░░░░░░░░ 40%            │  ← прогресс-бар
├─────────────────────────────────────────────────────┤
│  Чат                                     Память     │
└─────────────────────────────────────────────────────┘
```

### Элементы

| Элемент | Описание |
|---|---|
| Индикатор фазы | Цветная точка: жёлтый (planning), зелёный (execution), синий (validation), серый (paused) |
| Текст статуса | `status_text` из Task |
| Прогресс-бар | Заполнение по шагам: шаг 2/5 = 40%. В фазе planning — 0%, validation — 100% |
| Кнопка паузы [⏸] | Переводит в paused. В paused превращается в [▶] |
| Кнопка отмены [✕] | Переводит в cancelled с подтверждением ("Отменить задачу?") |

### Обновление данных

Фронтенд после каждого ответа LLM (в `onDone` стриминга) вызывает
`GET /api/tasks/{conversation_id}/active` и обновляет TaskPanel.

### Файл

Новый компонент `frontend/src/components/TaskPanel.tsx`.
API-клиент — `frontend/src/api/tasks.ts`.
Типы — добавляются в `frontend/src/types/index.ts`.

## 6. API эндпоинты

### Роутер: `modules/tasks/router.py`, prefix `/api/tasks`

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/{conversation_id}/active` | Активная задача диалога (или null) |
| `POST` | `/{conversation_id}` | Создать задачу (phase=planning) |
| `PUT` | `/{task_id}/transition` | Переход фазы `{ phase: string }` |
| `PUT` | `/{task_id}/plan` | Сохранить план `{ steps: [{description}] }` |
| `GET` | `/{conversation_id}/history` | Все задачи диалога (включая done/cancelled) |

### Кто вызывает

**Фронтенд (UI-действия):**
- `[⏸]` → `PUT /transition { phase: "paused" }`
- `[▶]` → `PUT /transition { phase: <previous_phase> }`
- `[✕]` → `PUT /transition { phase: "cancelled" }`

**Бэкенд (ChatService, автоматически):**
- Классификатор → "task" → `POST /{conversation_id}`
- План от LLM → `PUT /{task_id}/plan`
- Пользователь утвердил план → `PUT /transition { phase: "execution" }`
- "Шаг выполнен" → инкремент шага + возможный переход
- "Валидация пройдена" → `PUT /transition { phase: "done" }`

### Ответы

```
TaskOut {
    id: string
    conversation_id: string
    phase: string
    previous_phase: string | null
    title: string
    steps: [{description: string, status: string}] | null
    current_step: int
    status_text: string
    created_at: string
    updated_at: string
}
```

## 8. Интеграция с ChatService

### Изменения в `ChatService.__init__`

Добавляется `task_service: TaskService` через DI (аналогично `context_service`).

### Изменения в `process_message_stream`

```python
# 1. Проверяем активную задачу
active_task = await self._task_service.get_active(conversation_id)

if not active_task:
    # 2. Классификация
    msg_type = await self._task_service.classify_message(user_text)
    if msg_type == "task":
        active_task = await self._task_service.create(conversation_id, user_text)

# 3. Если есть задача — инжектируем фазовый промпт
if active_task:
    task_prompt = self._task_service.build_system_prompt(active_task)
    messages.insert(позиция_после_профиля, {"role": "system", "content": task_prompt})

# ... стриминг как сейчас ...

# 4. После ответа — анализ и автопереходы
if active_task:
    await self._task_service.process_llm_response(active_task, full_response)
```

### Порядок system messages

```
[0] инварианты (если есть)
[1] профиль (если есть)
[2] task phase prompt (если есть активная задача)
[3+] контекстные сообщения
→ merge в одно system-сообщение (требование GigaChat)
```

## 9. Тест-кейсы

### TC-01: Классификация — простой вопрос

**Сценарий:** Пользователь отправляет простой вопрос без активной задачи.
**Шаги:**
1. Создать новый диалог
2. Написать: "Как дела?" (или "Что такое Python?", "Переведи слово hello")
3. Дождаться ответа

**Ожидаемый результат:**
- TaskPanel НЕ появляется
- Агент отвечает напрямую, как обычный чат
- Никакой задачи не создаётся

---

### TC-02: Классификация — сложная задача

**Сценарий:** Пользователь ставит многошаговую задачу.
**Шаги:**
1. Создать новый диалог
2. Написать: "Напиши игру крестики-нолики на React. Выбери стек, составь план, потом реализуй"
3. Дождаться ответа

**Ожидаемый результат:**
- Появляется TaskPanel с жёлтым индикатором ● и текстом "Планирование: составляю план"
- Прогресс-бар на 0%
- Агент отвечает планом в формате JSON (список шагов)
- Агент спрашивает: "План готов. Утверждаете?"

---

### TC-03: Полный цикл — planning → execution → validation → done

**Сценарий:** Провести задачу через весь жизненный цикл.
**Шаги:**
1. Создать задачу (TC-02)
2. Написать: "Да, утверждаю" (или "Приступай")
3. Дождаться выполнения первого шага
4. Повторять подтверждение для каждого следующего шага (если нужно)
5. Дождаться перехода в validation
6. Дождаться завершения валидации

**Ожидаемый результат:**
- После утверждения: фаза → execution, индикатор → зелёный ●
- status_text обновляется: "Выполнение: шаг 1/N — ..."
- Прогресс-бар растёт с каждым шагом
- После последнего шага: фаза → validation, индикатор → синий ●
- После валидации: фаза → done, TaskPanel скрывается (или показывает "Задача завершена")

---

### TC-04: Запрещённый переход — planning → done

**Сценарий:** Попытка перескочить из planning сразу в done.
**Шаги:**

**Ожидаемый результат:**
- HTTP 400
- Сообщение об ошибке: переход planning → done не разрешён
- Фаза остаётся planning

### TC-06: Пауза из фазы planning

**Сценарий:** Поставить задачу на паузу во время планирования.
**Шаги:**
1. Создать задачу (фаза planning)
2. Нажать кнопку [⏸] в TaskPanel
3. Убедиться что фаза = paused
4. Нажать кнопку [▶]

**Ожидаемый результат:**
- При паузе: индикатор → серый ●, текст "На паузе (планирование)", кнопка [⏸] → [▶]
- При продолжении: фаза → planning, индикатор → жёлтый ●
- `previous_phase` корректно сохранён и использован

### TC-07: Пауза из фазы execution

**Сценарий:** Поставить задачу на паузу во время выполнения.
**Шаги:**
1. Довести задачу до execution (TC-03, шаги 1-2)
2. Нажать [⏸]
3. Убедиться что фаза = paused
4. Нажать [▶]
5. Написать "Продолжи"

**Ожидаемый результат:**
- При паузе: серый ●, прогресс-бар сохраняет позицию, `current_step` не сбрасывается
- При продолжении: фаза → execution, зелёный ●, тот же шаг
- Агент продолжает с текущего шага БЕЗ повторных объяснений и пересказа плана

**Сценарий:** Отменить задачу во время выполнения.
