# Тест-кейсы: Task FSM — конечный автомат состояний задачи

Сценарии для ручной проверки и записи видео.

**Подготовка:** запустить проект (`make build`), открыть в браузере `http://localhost:5173`.

---

## TC-01: Классификация — простой вопрос

**Цель:** Убедиться что простые вопросы НЕ запускают FSM.

**Шаги:**
1. Создать новый диалог (кнопка "Новый диалог")
2. Написать: "Как дела?"
3. Дождаться ответа

**Ожидаемый результат:**
- [ ] TaskPanel НЕ появляется
- [ ] Агент отвечает напрямую, как обычный чат
- [ ] Запрос `GET /api/tasks/{id}/active` возвращает `null`

---

## TC-02: Классификация — сложная задача

**Цель:** Убедиться что сложные задачи запускают FSM и переходят в planning.

**Шаги:**
1. Создать новый диалог
2. Написать: "Напиши игру крестики-нолики на React. Выбери стек, составь план, потом реализуй"
3. Дождаться ответа

**Ожидаемый результат:**
- [ ] Появляется TaskPanel с жёлтым индикатором ● 
- [ ] Текст статуса: "Планирование: ..."
- [ ] Прогресс-бар на 0%
- [ ] Агент отвечает планом (список шагов)
- [ ] Агент спрашивает подтверждение: "План готов. Утверждаете?"

---

## TC-03: Полный цикл — planning → execution → validation → done

**Цель:** Провести задачу через весь жизненный цикл.

**Шаги:**
1. Создать задачу (как в TC-02)
2. Написать: "Да, утверждаю" (или "Приступай")
3. Дождаться выполнения первого шага
4. Повторять подтверждение для каждого следующего шага (если нужно)
5. Дождаться перехода в validation
6. Дождаться завершения валидации

**Ожидаемый результат:**
- [ ] После утверждения: фаза → execution, индикатор → зелёный ●
- [ ] status_text обновляется: "Выполнение: шаг 1/N — ..."
- [ ] Прогресс-бар растёт с каждым шагом
- [ ] После последнего шага: фаза → validation, индикатор → синий ●
- [ ] После валидации: фаза → done, TaskPanel скрывается

---

## TC-04: Запрещённый переход — planning → done

**Цель:** Убедиться что нельзя перескочить из planning сразу в done.

**Шаги:**
1. Создать задачу (фаза planning)
2. Получить ID задачи из `GET /api/tasks/{conversation_id}/active`
3. Отправить: `PUT /api/tasks/{task_id}/transition` с телом `{ "phase": "done" }`

**Ожидаемый результат:**
- [ ] HTTP 400
- [ ] Сообщение об ошибке: переход planning → done запрещён
- [ ] Фаза остаётся planning (проверить через `GET /active`)

---

## TC-05: Запрещённый переход — planning → validation

**Цель:** Убедиться что нельзя перескочить из planning в validation.

**Шаги:**
1. Создать задачу (фаза planning)
2. Отправить: `PUT /api/tasks/{task_id}/transition` с телом `{ "phase": "validation" }`

**Ожидаемый результат:**
- [ ] HTTP 400
- [ ] Фаза остаётся planning

---

## TC-06: Пауза из фазы planning

**Цель:** Проверить паузу и продолжение на этапе планирования.

**Шаги:**
1. Создать задачу (фаза planning)
2. Нажать кнопку [⏸] в TaskPanel
3. Убедиться что фаза = paused
4. Нажать кнопку [▶]

**Ожидаемый результат:**
- [ ] При паузе: индикатор → серый ●
- [ ] Текст: "На паузе (планирование)"
- [ ] Кнопка [⏸] превращается в [▶]
- [ ] При продолжении: фаза → planning, индикатор → жёлтый ●

---

## TC-07: Пауза из фазы execution

**Цель:** Проверить паузу и продолжение на этапе выполнения.

**Шаги:**
1. Довести задачу до execution (TC-03, шаги 1-2)
2. Нажать [⏸]
3. Убедиться что фаза = paused
4. Нажать [▶]
5. Написать "Продолжи"

**Ожидаемый результат:**
- [ ] При паузе: серый ●, прогресс-бар сохраняет позицию
- [ ] `current_step` НЕ сбрасывается
- [ ] При продолжении: фаза → execution, зелёный ●, тот же шаг
- [ ] Агент продолжает с текущего шага БЕЗ повторных объяснений

---

## TC-08: Пауза из фазы validation

**Цель:** Проверить паузу и продолжение на этапе валидации.

**Шаги:**
1. Довести задачу до validation
2. Нажать [⏸]
3. Нажать [▶]

**Ожидаемый результат:**
- [ ] Аналогично TC-06/TC-07: возврат в validation, синий ●

---

## TC-09: Отмена задачи из execution

**Цель:** Проверить отмену задачи с подтверждением.

**Шаги:**
1. Довести задачу до execution
2. Нажать [✕]
3. Подтвердить отмену в диалоге подтверждения

**Ожидаемый результат:**
- [ ] Появляется подтверждение "Отменить задачу?"
- [ ] После подтверждения: фаза → cancelled, TaskPanel скрывается
- [ ] Новое сообщение можно отправить — классификатор снова работает

---

## TC-10: Отмена задачи — отказ от подтверждения

**Цель:** Проверить что отказ от подтверждения не отменяет задачу.

**Шаги:**
1. Довести задачу до execution
2. Нажать [✕]
3. Нажать "Отмена" в подтверждении

**Ожидаемый результат:**
- [ ] Фаза НЕ меняется, задача продолжается
- [ ] TaskPanel остаётся без изменений

---

## TC-11: Продолжение после паузы — без повторных объяснений

**Цель:** Проверить что агент не пересказывает контекст после паузы.

**Шаги:**
1. Создать задачу, довести до execution шаг 3/5
2. Нажать [⏸]
3. Нажать [▶]
4. Написать: "Продолжи"

**Ожидаемый результат:**
- [ ] Агент НЕ пишет "Ранее мы делали...", "Напомню план..."
- [ ] Агент сразу выполняет шаг 3
- [ ] status_text: "Выполнение: шаг 3/5 — ..."

---

## TC-12: validation → execution (доработка)

**Цель:** Проверить возврат из валидации в выполнение при обнаружении проблемы.

**Шаги:**
1. Довести задачу до validation
2. Если LLM находит проблему — он пишет "Требуется доработка"

**Ожидаемый результат:**
- [ ] Фаза → execution
- [ ] Индикатор → зелёный ●
- [ ] Агент возвращается к доработке

---

## TC-13: UI — корректность отображения TaskPanel

**Цель:** Проверить визуальные элементы TaskPanel во всех фазах.

**Шаги:**
1. Провести задачу через все фазы, на каждой проверить таблицу:

**Ожидаемый результат:**

| Фаза | Цвет точки | Текст | Прогресс | Кнопки |
|---|---|---|---|---|
| planning | жёлтый | "Планирование: ..." | 0% | [⏸] [✕] |
| execution | зелёный | "Выполнение: шаг X/Y — ..." | X/Y * 100% | [⏸] [✕] |
| validation | синий | "Валидация: проверяю результат" | 100% | [⏸] [✕] |
| paused | серый | "На паузе ({previous})" | сохранён | [▶] [✕] |
| done | — | TaskPanel скрывается | — | — |
| cancelled | — | TaskPanel скрывается | — | — |

- [ ] Все цвета соответствуют таблице
- [ ] Текст статуса обновляется при каждом переходе
- [ ] Прогресс-бар корректно заполняется

---

## TC-14: Новая задача после завершения предыдущей

**Цель:** Проверить что после done можно создать новую задачу.

**Шаги:**
1. Завершить задачу (done)
2. Написать новую сложную задачу: "Теперь добавь авторизацию"

**Ожидаемый результат:**
- [ ] Классификатор срабатывает заново (активной задачи нет)
- [ ] Создаётся новая задача, фаза planning
- [ ] TaskPanel снова появляется
- [ ] Предыдущая задача видна через `GET /api/tasks/{conversation_id}/history`
